# AAM META CORE PASSPORT

## 1. Purpose
AAM Meta Core служит фундаментальным слоем (L0) для всей архитектуры `AAM-Language-Kernels`. Его цель — обеспечить универсальный логический и методологический базис, гарантирующий воспроизводимость, детерминизм и семантическую точность для любого языкового или доменного ядра, подключенного к системе.

## 2. Invariants
Система базируется на пяти инвариантах, нарушение которых приводит к аннулированию статуса «AAM-compliant»:
* **[INV-01] Hierarchical Integrity:** Любой анализ строго структурирован сверху вниз (от SOURCE к EXECUTION).
* **[INV-02] Semantic Primacy:** Онтологическая суть объекта имеет приоритет над лингвистической оболочкой.
* **[INV-03] 720° Stabilization:** Запрещено делать выводы до завершения полного цикла (Синтаксис → Семантика → Стабилизация).
* **[INV-04] Determinism:** SOL-примитивы являются конечным уровнем верификации любого аналитического вывода.
* **[INV-05] Reproducibility:** При одинаковых [CONSTRAINT] и входных данных система обязана генерировать идентичный результат.

## 3. SOL Primitives (Semantic Object Language)
Все процессы описываются через 6 базовых примитивов:
* **[SOURCE]:** Точка инициации, источник ресурса или сигнала.
* **[CONSTRAINT]:** Физический, логический или временной предел/барьер.
* **[FLOW]:** Движение ресурса, данных или энергии через систему.
* **[TRANSFORM]:** Преобразование параметров объекта при прохождении через узлы.
* **[VERIFY]:** Механизм сравнения STATE_actual с STATE_expected.
* **[DISSIPATE]:** Рассеивание, завершение процесса, отказ или выход за пределы системы.

## 4. Constraint-First Method
Философия анализа строится на приоритете ограничений:
1. Идентификация [CONSTRAINT].
2. Определение допустимого пространства решений.
3. Поиск оптимального [FLOW] внутри границ.
4. Верификация через [VERIFY].

## 5. State Model
Любой объект системы существует в двух состояниях:
* **STATE_expected:** Инженерный/логический эталон (замысел).
* **STATE_actual:** Наблюдаемые параметры в текущем контексте.

## 6. TDI Model (Tensor Divergence Index)
Количественная мера ошибки вычисляется по формуле:
`TDI = |STATE_actual - STATE_expected| / CONSTRAINT_limit`
Если TDI > 1.0, система фиксирует отказ [DISSIPATE] и требует принудительной стабилизации.

## 7. Reproducibility Rules
* Исключение вероятностных галлюцинаций через строгое следование SOL-примитивам.
* Фиксация всех промежуточных векторов состояния при выполнении 720°-цикла.
* Использование только стандартизированных словарей примитивов.

## 8. Dependency Rules
* L0 является корневым узлом: не имеет зависимостей.
* Любое изменение в L0 требует обязательного аудита всех L1-L5 модулей.
* Недопустимо создание циклических зависимостей между ядрами (L2).

## 9. Versioning Policy
* **Major (vX.0):** Изменение инвариантов или SOL-примитивов (нарушение обратной совместимости).
* **Minor (v1.X):** Добавление новых языковых ядер (L2) или доменных пакетов (L3).
* **Patch (v1.X.Y):** Исправление ошибок в существующей логике.
