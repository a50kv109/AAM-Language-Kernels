# LANGUAGE KERNEL TEMPLATE (v1.1.2)

## 1. Kernel Identification
* **Kernel ID:** `[LANGUAGE_NAME]-Core`
* **Version:** 1.0.0
* **Status:** Level 0 (Prototype)
* **Architecture:** AAM-V1 / SOL-ECC-1

## 2. Purpose
Универсальный стандарт для создания языковых ядер в рамках AAM-Language-Kernels. Шаблон гарантирует, что любое новое языковое ядро будет обладать идентичной внутренней структурой, поддерживать SOL-примитивы и обеспечивать бесшовную интеграцию с доменными пакетами (L3).

## 3. Reference Architecture
Система функционирует согласно строгой иерархии:
1. **L0: Meta Core (AAM)** — Базовые инварианты и логика.
2. **L1: ECC-1 Kernel** — Движок причинно-следственного мышления.
3. **L2: Language Kernel** — Лингвистическая реализация (данный шаблон).
4. **L3: Domain Packs** — Прикладные модули (Med, Fin, Engineering, и др.).

## 4. Analysis Axes (Universal Standard)
Каждое ядро обязано реализовывать 9 осей анализа:

| Axis | Name | Description | Function |
| :--- | :--- | :--- | :--- |
| **Axis-1** | Phonetic | Фонетика, просодия, тональность | Стабилизация звуковой формы |
| **Axis-2** | Morphological | Морфемы, аффиксы, словообразование | Выделение атомарных единиц |
| **Axis-3** | Syntactic | Порядок слов, дерево связей | Определение структуры связи |
| **Axis-4** | Semantic | Смыслы, многозначность | Извлечение смысла |
| **Axis-5** | Pragmatic | Интенции, вежливость, контекст | Анализ цели высказывания |
| **Axis-6** | Cultural | Идиомы, табу, история | Коррекция семантического дрейфа |
| **Axis-7** | Cognitive | Ментальные модели, предположения | Анализ глубинных моделей |
| **Axis-8** | SOL Mapping | Привязка к [SOURCE], [CONSTRAINT] и др. | Преобразование в SOL |
| **Axis-9** | Domain Interface | Подключение доменов (L3) | Интеграция с прикладным API |

## 6. Domain Integration
Механизм подключения доменных пакетов (L3):
* Интерфейс для обмена данными с `Domain Packs`.
* Использование единого протокола примитивов (см. `primitives/`).

## 7. Reference Examples
Каждое ядро должно содержать не менее 25 эталонных примеров в каталоге `examples/`:
* 10 базовых примеров.
* 10 сложных примеров (с неоднозначностью или идиомами).
* 5 доменных примеров (Med/Fin/Eng).
* **Структура примера:** Input → Analysis (9 осей) → Output.

## 8. Cross-Language Compatibility
Механизм сравнения ядер:
* Проверка сохранения смысла при переносе через SOL-примитивы.
* Соблюдение иерархических зависимостей при кросс-языковом анализе.
* Валидация прагматики и SOL-маппинга в разных лингвистических системах.

## 9. Benchmark Requirements
Минимальный набор тестов для верификации нового ядра (см. `benchmarks/`):
1. **Portability:** Успешный запуск в 3 разных LLM-средах.
2. **Consistency:** Идентичность вывода при 3-кратном прогоне (TDI=0).
3. **Hierarchy Preservation:** Соблюдение иерархических зависимостей.
4. **Cultural Compliance:** Корректная обработка маркеров из `cultural/` слоя.
5. **Cross-Language Compatibility Test:** Успешное сравнение с эталонным ядром.

## 10. Failure Modes
Типовые ошибки, подлежащие контролю (см. `failure_modes.md`):
1. Потеря иерархического контекста.
2. Буквальный перевод идиоматических выражений.
3. Ошибка суперпозиции смыслов.
4. Нарушение инвариантов L0.
5. Игнорирование прагматического контекста.
6. Ошибки интерпретации грамматических маркеров.
7. Некорректная обработка культурных табу.
8. Семантический дрейф при конвертации.
9. Отсутствие стабилизации (вывод до 720°-цикла).
10. Несоответствие вывода формату [FINDING] + [RISK] + [RECOMMENDATION].

## 11. Reference Implementation Requirements
Для получения статуса Level 3 ядро должно:
* Реализовывать все 9 осей.
* Иметь полный паспорт.
* Иметь не менее 25 эталонных примеров.
* Пройти Benchmark Suite.
* Пройти Cross-Language Compatibility Test.
* Иметь документированные Failure Modes.

## 12. Minimum Repository Structure
Каждое новое ядро должно содержать:
* `passport.md`
* `axes/`
* `semantics/`
* `examples/`
* `tests/`
* `failure_modes.md`
* `benchmarks.md`

## 13. Dependencies
* **L0:** AAM Meta Core.
* **L1:** ECC-1 Kernel.

## 14. Versioning Policy
* **Major:** Изменение набора осей анализа.
* **Minor:** Добавление новых правил в Axis-8 или Axis-9.
* **Patch:** Исправление Failure Modes или обновление документации.
